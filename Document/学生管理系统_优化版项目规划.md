# 学生管理系统 — 优化版项目规划

## 1. 项目目标与范围

本项目建设一个面向高校或培训机构的学生管理系统，覆盖院系、专业、班级、学生、教师、课程、成绩、用户认证与权限控制等核心业务。系统采用前后端分离架构：后端使用 Spring Boot + MyBatis-Plus + Spring Security + JWT，前端使用 Vue 3 + Element Plus + Pinia + Axios。

### 1.1 核心目标

- 实现基础教务数据的增删改查与分页检索。
- 实现基于角色的访问控制，区分管理员、教师、学生三类用户。
- 支持成绩录入、查询、统计分析。
- 提供仪表盘，展示学生、教师、课程、成绩等关键指标。
- 保证数据完整性、接口规范性、权限安全性和后续扩展能力。

### 1.2 角色定义

| 角色 | 说明 |
|---|---|
| ADMIN | 系统管理员，拥有全部业务数据管理权限。 |
| TEACHER | 教师用户，可查看基础教务信息，管理本人所授课程的成绩。 |
| STUDENT | 学生用户，仅可查看本人信息、本人课程与成绩。 |

## 2. 业务实体与关系设计

### 2.1 核心实体

| 实体 | 关键字段 | 主要关系 |
|---|---|---|
| 院系 department | id, name, dean, phone | 1:N → major；1:N → teacher |
| 专业 major | id, name, department_id, duration | N:1 → department；1:N → clazz |
| 班级 clazz | id, name, major_id, grade, advisor | N:1 → major；1:N → student |
| 学生 student | id, student_no, name, gender, birth_date, clazz_id, phone | N:1 → clazz；N:N → course，通过 score 关联 |
| 教师 teacher | id, teacher_no, name, gender, department_id, title, phone | N:1 → department；1:N → course |
| 课程 course | id, course_no, name, credit, hours, teacher_id | N:1 → teacher；N:N → student，通过 score 关联 |
| 成绩 score | id, student_id, course_id, score, semester | N:1 → student；N:1 → course |
| 用户 sys_user | id, username, password, role, ref_id, status | 认证实体，ref_id 关联 student 或 teacher |

### 2.2 关系约束建议

- department 删除前必须校验是否存在 major 或 teacher。
- major 删除前必须校验是否存在 clazz。
- clazz 删除前必须校验是否存在 student。
- teacher 删除前必须校验是否存在 course。
- course 删除前必须校验是否存在 score。
- student 删除时建议同步禁用或删除对应 sys_user。
- score 中同一学生、同一课程、同一学期只能有一条成绩记录。

## 3. 数据库设计优化

### 3.1 通用字段

所有业务表建议统一包含以下字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| id | BIGINT | 主键，自增 |
| create_time | DATETIME | 创建时间 |
| update_time | DATETIME | 更新时间 |
| deleted | TINYINT | 逻辑删除标记，0 未删除，1 已删除 |

| 字段 | 类型 | 说明 |

### 4.2 后端分层规范

| 层级 | 职责 |
|---|---|
| Controller | 参数接收、权限注解、调用 Service，不写业务逻辑。 |
| Service | 业务校验、事务控制、权限范围过滤。 |
| Mapper | 数据访问，仅处理 SQL 与查询条件。 |
| DTO/VO | 隔离前端请求、响应与数据库实体。 |
| Common | 统一响应、异常、分页、枚举、常量。 |

### 4.3 统一响应格式

```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```

分页响应：

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "page": 1,
    "size": 10,
    "total": 100,
    "records": []
  }
}
```

### 4.4 全局异常分类

| 异常类型 | HTTP 状态码 | 业务码 | 说明 |
|---|---:|---:|---|
| 参数校验失败 | 400 | 40001 | DTO 参数非法 |
| 未认证 | 401 | 40101 | Token 缺失、过期或非法 |
| 无权限 | 403 | 40301 | 角色或数据权限不足 |
| 资源不存在 | 404 | 40401 | 查询目标不存在 |
| 业务冲突 | 409 | 40901 | 唯一约束冲突、存在下级数据不可删除 |
| 系统异常 | 500 | 50001 | 未预期异常 |

## 5. API 设计优化

### 5.1 通用规范

- API 前缀统一使用 `/api`。
- 列表查询统一支持 `page`、`size`、`keyword`。
- 删除操作优先使用逻辑删除。
- 新增、修改接口使用 Request DTO。
- 详情、列表接口使用 Response VO，避免直接返回实体。
- 所有需要登录的接口必须经过 JWT 认证。

### 5.2 认证模块

| 方法 | 路径 | 权限 | 说明 |
|---|---|---|---|
| POST | /api/auth/login | 公开 | 登录，返回 token 与用户信息 |
| GET | /api/auth/info | 登录用户 | 获取当前用户信息 |
| POST | /api/auth/logout | 登录用户 | 前端清理 token，后端可选黑名单 |
| PUT | /api/auth/password | 登录用户 | 修改当前用户密码 |

### 5.3 基础数据模块

院系、专业、班级、学生、教师、课程、成绩均采用统一 CRUD 风格：

| 方法 | 路径 | 说明 |
|---|---|---|
| GET | /api/{resources} | 分页查询 |
| GET | /api/{resources}/{id} | 查询详情 |
| POST | /api/{resources} | 新增 |
| PUT | /api/{resources}/{id} | 修改 |
| DELETE | /api/{resources}/{id} | 删除 |

资源路径建议：

| 模块 | 路径 |
|---|---|
| 院系 | /api/departments |
| 专业 | /api/majors |
| 班级 | /api/classes |
| 学生 | /api/students |
| 教师 | /api/teachers |
| 课程 | /api/courses |
| 成绩 | /api/scores |

> 建议将原文中的 `/api/clazzes` 调整为 `/api/classes`，对前端和接口使用者更直观。

### 5.4 成绩扩展接口

| 方法 | 路径 | 权限 | 说明 |
|---|---|---|---|
| GET | /api/scores | ADMIN / TEACHER / STUDENT | 分页查询，按角色自动过滤数据范围 |
| POST | /api/scores | ADMIN / TEACHER | 新增成绩 |
| PUT | /api/scores/{id} | ADMIN / TEACHER | 修改成绩 |
| DELETE | /api/scores/{id} | ADMIN | 删除成绩，教师默认不允许删除 |
| GET | /api/scores/statistics | ADMIN / TEACHER / STUDENT | 成绩统计 |
| POST | /api/scores/import | ADMIN / TEACHER | 批量导入成绩，可选 |
| GET | /api/scores/export | ADMIN / TEACHER | 导出成绩，可选 |

### 5.5 仪表盘接口

| 方法 | 路径 | 权限 | 说明 |
|---|---|---|---|
| GET | /api/dashboard/overview | 登录用户 | 统计卡片数据 |
| GET | /api/dashboard/score-distribution | 登录用户 | 成绩分布 |
| GET | /api/dashboard/department-students | ADMIN / TEACHER | 院系学生人数统计 |
| GET | /api/dashboard/recent-scores | ADMIN / TEACHER | 最近成绩记录 |

## 6. 权限设计优化

### 6.1 权限矩阵

| 模块 | ADMIN | TEACHER | STUDENT |
|---|---|---|---|
| 院系管理 | CRUD | R | - |
| 专业管理 | CRUD | R | - |
| 班级管理 | CRUD | R | - |
| 学生管理 | CRUD | R | R，本人 |
| 教师管理 | CRUD | R，本人 | - |
| 课程管理 | CRUD | R，本人所授课程 | R，已选或相关课程 |
| 成绩管理 | CRUD | C/R/U，本人所授课程 | R，本人成绩 |
| 用户管理 | CRUD | 修改本人密码 | 修改本人密码 |
| 仪表盘 | 全局统计 | 所授课程统计 | 本人统计 |

### 6.2 数据权限规则

#### ADMIN

- 可访问全部数据。
- 可新增、修改、删除所有业务数据。
- 可创建、禁用、重置用户密码。

#### TEACHER

- 可查看院系、专业、班级、学生基础信息。
- 只能新增或修改本人所授课程的成绩。
- 不能删除成绩，除非业务明确授权。
- 只能查看本人教师信息。
- 不能修改课程归属教师。

#### STUDENT

- 只能查看本人学生信息。
- 只能查看本人成绩。
- 只能修改本人密码。
- 不能访问管理类写接口。

### 6.3 前后端权限配合

- 前端基于角色控制菜单与按钮显示。
- 后端必须进行接口级权限校验，不能只依赖前端隐藏。
- 涉及本人、本人所授课程等数据范围的接口，需要在 Service 层二次校验。

- `common/`
- `config/`

| 院系管理 | 支持分页、搜索、新增、编辑、删除；存在下级数据时禁止删除。 |
| 专业管理 | 支持按院系筛选；院系不存在时禁止新增。 |
| 班级管理 | 支持按专业、年级筛选；专业不存在时禁止新增。 |
| 学生管理 | 学号唯一；支持按班级、姓名、学号搜索。 |
| 教师管理 | 工号唯一；支持按院系、姓名搜索。 |
| 课程管理 | 课程号唯一；授课教师必须存在。 |
| 成绩管理 | 同一学生、课程、学期成绩唯一；成绩范围 0-100。 |
| 权限控制 | 三类角色只能访问授权范围内的数据与操作。 |
| 仪表盘 | 不同角色展示对应统计范围。 |

### 9.2 非功能验收

| 类型 | 标准 |
|---|---|
| 接口规范 | 所有接口统一响应结构。 |
| 安全性 | 密码 BCrypt 加密；JWT 过期可控；后端强制权限校验。 |
| 可维护性 | Controller 不写复杂业务逻辑；DTO 与 Entity 分离。 |
| 数据完整性 | 唯一约束、外键关联或业务校验完整。 |
| 用户体验 | 表单校验、加载状态、错误提示、删除确认完整。 |

## 10. 风险点与规避方案

| 风险 | 影响 | 规避方案 |
|---|---|---|
| 只做前端权限控制 | 用户可绕过页面直接调用接口 | 后端必须使用角色权限和数据权限校验。 |
| 直接返回 Entity | 暴露敏感字段，如 password | 使用 Response VO，用户表永不返回 password。 |
| 删除基础数据导致脏数据 | 数据不一致 | 删除前检查下级数据，优先逻辑删除。 |
| 成绩重复录入 | 统计结果错误 | 添加唯一约束：student_id + course_id + semester。 |
| JWT 过期处理不统一 | 用户体验差 | Axios 响应拦截器统一处理 401。 |
| 后期再接权限 | 返工成本高 | 认证与权限模块提前完成。 |
| 接口字段与前端命名不一致 | 联调成本高 | 提前定义 DTO 和接口文档。 |
